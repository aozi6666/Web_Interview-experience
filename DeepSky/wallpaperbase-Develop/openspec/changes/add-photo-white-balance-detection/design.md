## Context

当前项目的照片/壁纸能力主要集中在渲染层展示、详情编辑和本地配置同步，现有流程没有针对静态图片做画面质量分析。白平衡检测属于轻量级单图分析能力，输入通常来自预览图 URL、本地图片路径或资源详情图，因此更适合在渲染层完成一次性计算并直接反馈到界面，而不是引入新的后端链路。

## Goals / Non-Goals

**Goals:**

- 为单张照片提供可复用的白平衡检测入口。
- 输出统一的检测结果模型，至少覆盖正常、偏暖、偏冷、无法判定和检测失败。
- 让预览卡片或详情面板能够以一致方式展示检测结果和处理中状态。
- 保持实现简单，可在后续扩展到曝光、清晰度等其他画质分析能力。

**Non-Goals:**

- 不引入云端图像分析服务或新增后端 API。
- 不在本次设计中覆盖视频白平衡分析。
- 不追求专业摄影级色彩校准，只提供面向产品提示的轻量判断。
- 不强制持久化检测结果到远端资源数据。

## Decisions

### 1. 在渲染层执行白平衡检测

选择在渲染层通过图片解码和 Canvas 像素采样完成分析。

原因：

- 当前照片预览与详情展示已经位于渲染层，接入成本最低。
- 白平衡检测是读取型分析，不需要主进程特权。
- 可以直接复用现有组件状态管理，避免增加 IPC 或服务端接口。

备选方案：

- 主进程分析：适合统一文件访问，但会增加 IPC、序列化和错误处理复杂度。
- 云端分析：结果更容易持久化，但会引入网络依赖、成本和隐私问题。

### 2. 使用轻量启发式算法，而不是引入模型推理

选择基于采样像素的 RGB 通道统计与中性灰偏移量来判断照片偏暖、偏冷或近似平衡，并输出有限的置信度或偏移指标。

原因：

- 白平衡提示不需要高成本模型即可完成第一版产品目标。
- 项目虽然已有 `tfjs` 依赖，但为此单独建立模型推理链路会显著提高复杂度。
- 启发式规则更容易调参与测试，也更容易解释结果。

备选方案：

- 机器学习模型分类：准确率可能更高，但对样本、性能和调试要求更高。
- 仅依赖 EXIF 元数据：实现简单，但很多图片没有可用元数据，且不能反映最终视觉观感。

### 3. 统一检测结果结构并支持缓存

选择定义统一结果结构，例如：

- `status`: `balanced | warm | cool | unknown | error`
- `score`: 用于表示偏移强度或置信度
- `reason`: 可选，用于说明失败或无法判定原因

同时以图片唯一标识（如资源 ID、URL 或本地路径）做内存级缓存，避免重复分析同一张图。

原因：

- 结果结构稳定后，卡片、详情和后续其他入口都能共用。
- 内存缓存足以覆盖当前会话内重复浏览场景，复杂度远低于持久化。

备选方案：

- 每次进入页面重新计算：实现最简单，但重复开销较高，且 UI 容易闪烁。
- 将结果持久化到资源配置：可减少重复计算，但需要评估数据模型演进和失效策略。

### 4. 展示策略采用“异步非阻塞提示”

检测在图片可用后异步触发，UI 展示“分析中”占位，不阻塞预览、详情查看或设为壁纸操作。若分析失败，界面显示可理解的失败状态，而不是隐藏该功能。

原因：

- 该能力本质是质量提示，不应影响核心浏览和应用流程。
- 非阻塞方式更符合现有界面的轻交互风格。

备选方案：

- 在应用前强制检测并阻断：能提高一致性，但会损害使用流畅度。
- 仅在详情页手动触发：实现更保守，但用户感知弱。

## Risks / Trade-offs

- [启发式算法存在误判] → 通过阈值调优、`unknown` 状态和文案弱化来降低误导。
- [跨域或无效图片导致 Canvas 读取失败] → 在分析前统一做图片可用性校验，并返回 `error` 或 `unknown`。
- [首次分析带来轻微性能开销] → 控制采样分辨率并加入内存缓存，避免全尺寸逐像素分析。
- [不同入口展示不一致] → 使用统一状态枚举和展示映射，减少各组件自行解释结果。
