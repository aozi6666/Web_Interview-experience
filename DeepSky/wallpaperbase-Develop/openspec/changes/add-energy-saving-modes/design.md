## Context

当前代码已经有一套壁纸显示模式链路：

- 共享类型 `WallpaperDisplayMode` 当前包含 `Interactive`、`EnergySaving`、`StaticFrame`。
- 渲染侧 `SystemStatusContext` 会根据 `UEState` 与全屏覆盖状态推导当前显示模式，并负责在 `Interactive` / `EnergySaving` / `StaticFrame` 之间切换视频壁纸与互动壁纸。
- UI 已存在 `WallpaperModeSwitcher`，可供用户手动切换模式。
- 主进程已有全局鼠标钩子与桌面鼠标事件链路，可作为“桌面连续点击 5 次”触发互动模式的基础。
- 全屏检测已经区分 `red` / `orange` / `yellow` 等覆盖等级，但目前渲染侧仅把“存在覆盖”统一映射到 `StaticFrame`。

这次变更是一个跨主进程、渲染进程、共享类型和模式策略的收敛改动，需要补齐默认模式、手动唤醒互动、桌面鼠标手势，以及更细粒度的全屏节能等级。

## Goals / Non-Goals

**Goals:**

- 将默认显示模式统一为 `EnergySaving`。
- 支持用户通过聊天框旁按钮显式打开或关闭互动模式。
- 支持用户在桌面上连续点击 5 次唤醒互动模式。
- 支持全屏检测分级：红色进入 `ExtremeLow`，其余全屏覆盖进入 `StaticFrame`。
- 明确自动节能、手动互动和全屏降级之间的优先级与恢复行为。

**Non-Goals:**

- 不改变壁纸资源格式、UE/WE 资源内容或具体壁纸条目数据。
- 不引入新的外部依赖。
- 不重写现有全屏检测算法，只消费现有颜色级别结果。
- 不在本次设计中扩展更多手势类型，只支持“桌面连续 5 次点击”。

## Decisions

### 1. 引入“用户意图 + 环境约束”两层模式决策

决策：

- 保留当前“实际显示模式”概念，但新增一个持久的“用户偏好/意图”层，至少区分：
  - 默认节能态
  - 手动开启互动
  - 手动关闭互动
- 最终显示模式由“用户意图”与“环境约束”共同计算：
  - 无全屏覆盖时：手动互动优先输出 `Interactive`，否则输出 `EnergySaving`
  - 全屏红色时：无条件输出 `ExtremeLow`
  - 其他全屏覆盖时：无条件输出 `StaticFrame`

原因：

- 现在的模式推导更多依赖运行时状态，缺少“用户刚刚主动要求互动”这一层语义。
- 把“用户意图”和“环境约束”拆开后，才可以在全屏结束后恢复到正确模式，而不是丢失用户手动切换的上下文。

备选方案：

- 直接把按钮和鼠标触发都映射成一次性 `changeUEState('3D')`。
  - 放弃原因：这种方式无法表达“用户当前是否处于手动互动会话”，全屏覆盖结束后也难以恢复到正确状态。

### 2. 扩展 `WallpaperDisplayMode`，但尽量不扩大底层 UE 状态枚举

决策：

- 在共享显示模式上新增 `ExtremeLow`。
- 底层 UE 运行态仍优先复用现有 `3D` / `EnergySaving`，由上层模式策略决定视频播放、静帧显示或极低功耗行为。

原因：

- `WallpaperDisplayMode` 是用户可见语义，适合表达 `ExtremeLow`。
- `UEState` 及多处主进程逻辑目前只认 `3D` / `EnergySaving`，若强行扩成四态，会放大改动面。
- `ExtremeLow` 更像是“EnergySaving` 下的更强降级策略”，不一定需要新增底层 UE 协议枚举。

备选方案：

- 同步扩展 `UEState`、`DisplayCoordinator`、`FullscreenDetectorManager` 等所有主流程为四态。
  - 放弃原因：侵入面大，而且当前需求本质是显示策略分级，不一定要求底层运行时协议同步升级。

### 3. 聊天框按钮复用现有模式切换组件能力，补充“互动开关”语义

决策：

- 在聊天框旁按钮或其所属组件中复用 `WallpaperModeSwitcher` / `useWallpaperDisplayMode` 的现有入口。
- 互动按钮行为定义为：
  - 当前非互动时，点击进入 `Interactive`
  - 当前已互动时，点击关闭互动，回到 `EnergySaving`

原因：

- 现有模式切换 UI 已有基础能力，优先复用而不是并行增加第二套状态入口。
- “按钮可手动关闭互动”是对当前模式切换语义的明确化，不需要引入新的独立控制器。

备选方案：

- 增加一个单独的“互动开关按钮”，与现有模式切换器并存。
  - 放弃原因：容易形成两套控制源，增加同步复杂度与 UI 认知负担。

### 4. 桌面五连击由主进程检测，渲染侧只消费“唤醒互动”事件

决策：

- 在主进程现有的系统鼠标截获与转发链路中检测“桌面连续 5 次左键点击”，优先复用 `MouseEventForwarder` 作为入口。
- 检测条件包括：
  - 点击目标命中桌面壁纸层，而不是任意其它非透明应用窗口
  - 不能只看“当前在桌面屏幕”，还要确认命中的顶层窗口确实属于桌面壁纸承载层
  - 仅统计左键单击
  - 使用短时间窗口累计，超时或切出桌面后重置
- 一旦命中，主进程发出统一 IPC/事件，交由模式策略层切到手动互动。

原因：

- 该手势发生在桌面环境，主进程最适合感知，避免依赖渲染窗口焦点。
- 把主进程职责限定为“识别手势 + 发事件”，把真正的模式切换保持在统一策略层，避免分叉。
- 现有鼠标操作已经由 Electron 主进程截获系统鼠标消息再转发，直接在这条链路增加壁纸层命中检测，复用度最高，也最容易保证与现有桌面交互语义一致。

备选方案：

- 在渲染侧通过 DOM 事件统计连击。
  - 放弃原因：用户是在桌面上点击，不保证渲染窗口处于前台，事件会丢失。
- 只要 `WindowFromPoint` 命中任意窗口就视为桌面点击。
  - 放弃原因：这会把普通非透明应用窗口也算进五连击，和“只在桌面壁纸层触发”的产品语义冲突。

### 5. 全屏降级优先级高于手动互动，但结束后恢复先前用户意图

决策：

- 当存在全屏覆盖时：
  - `red` -> `ExtremeLow`
  - `orange` / `yellow` -> `StaticFrame`
- 该规则优先级高于按钮或五连击触发的互动模式。
- 覆盖结束后，不直接固定回 `EnergySaving`，而是重新依据用户意图计算，确保手动互动可恢复。

原因：

- 全屏场景的节能保护属于环境硬约束，必须盖过用户触发的互动模式。
- 但如果用户是在全屏前主动进入互动，覆盖结束后回到互动更符合“手动意图持续有效”的预期。

备选方案：

- 全屏结束后一律回到 `EnergySaving`。
  - 放弃原因：会丢失用户手动切到互动的意图，体验上不连续。

## Risks / Trade-offs

- [风险] `ExtremeLow` 只扩展显示模式、不扩展底层 UE 状态，可能让模式名与实际底层状态不是一一对应。 → Mitigation：在设计中明确 `ExtremeLow` 属于上层显示策略，文档和类型命名保持一致，并在实现中集中封装映射。
- [风险] 桌面五连击若只做全局计数、不校验是否命中壁纸层，可能把普通应用点击误判为桌面手势。 → Mitigation：在主进程鼠标转发入口增加壁纸层命中检测，仅当顶层命中对象属于桌面壁纸承载层时计数。
- [风险] 全屏降级与手动互动同时存在时容易出现状态抖动。 → Mitigation：统一通过单一策略层计算最终模式，禁止各入口直接绕过策略层改底层状态。
- [风险] 现有依赖 `WallpaperDisplayMode` 的 UI 与 hooks 在新增 `ExtremeLow` 后会出现未覆盖分支。 → Mitigation：统一扫描 `WallpaperDisplayMode` 的 `switch/if` 使用点，补齐分支与文案。

## Migration Plan

- 本次变更不涉及数据迁移。
- 采用增量落地：
  1. 先扩展共享模式枚举和模式策略层。
  2. 再接入聊天框按钮与桌面五连击入口。
  3. 最后接入全屏红色/其他覆盖的细分降级和恢复逻辑。
- 若上线后发现 `ExtremeLow` 行为不稳定，可先退回为“红色也映射到 `StaticFrame`”，保留其余结构不变。

## Open Questions

- `ExtremeLow` 的具体执行语义是否仅为“暂停/不恢复视频”，还是需要额外黑帧、低帧率或彻底禁用某些渲染链路？本设计先把它定义为一种显示模式，并要求实现阶段明确其底层行为。
- 桌面五连击的判定时间窗口是否需要产品侧固定值；若无额外要求，建议实现时采用短窗口并保持常量集中定义。
