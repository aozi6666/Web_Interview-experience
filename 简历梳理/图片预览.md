# ClawX 附件预览：本地缓存与异步补齐（源码学习导览）

这份笔记专门用来帮助你在源码里快速找到：**附件预览如何先走本地缓存、哪些情况下会触发异步补齐、以及前端如何根据 `preview` 的有无进行渲染**。

---

## 1. 实现入口一览（从“发送”到“渲染”的链路）

1. 发送前写入本地缓存（避免竞态导致 preview 丢失）
   - `src/stores/chat/runtime-send-actions.ts`
   - 关键函数：`sendMessage`
   - 关键逻辑：在 IPC/RPC 调用前对图片 attachments 执行 `upsertImageCacheEntry(a.stagedPath, ...)`

2. 本地缓存读写 + 从历史消息中恢复 `_attachedFiles` + 解析附件引用
   - `src/stores/chat/helpers.ts`
   - 关键函数/模块：
     - `IMAGE_CACHE_KEY` / `IMAGE_CACHE_MAX`
     - `loadImageCache()` / `saveImageCache()`
     - `_imageCache`（模块级 Map）
     - `upsertImageCacheEntry(filePath, file)`
     - `enrichWithCachedImages(messages)`（历史恢复：从文本提取 `[media attached: ...]`/绝对路径，然后优先从缓存取 preview）
     - `loadMissingPreviews(messages)`（异步补齐：只为缺失 preview 的图片/文件请求缩略图）

3. 历史加载完成后触发“异步补齐”
   - `src/stores/chat/history-actions.ts`
   - 关键点：`loadHistory()` 内 `applyLoadedMessages(...)` 之后调用：
     - `loadMissingPreviews(finalMessages).then(...)`

4. UI 渲染：有 `file.preview` 就显示缩略图；没有就显示占位
   - `src/pages/Chat/ChatMessage.tsx`
   - 关键点：遍历 `message._attachedFiles`
     - 如果 `file.mimeType.startsWith('image/')`：
       - `file.preview` 存在 -> `ImageThumbnail src={file.preview}`
       - `file.preview` 为空 -> 显示占位卡片

5.（对照/旧实现）同逻辑在 `src/stores/chat.ts` 里也存在
   - `src/stores/chat.ts`
   - 与 helpers.ts 的差异：后端调用形式不同（helpers.ts 用 IPC：`media:getThumbnails`；chat.ts 用 HTTP：`/api/files/thumbnails`）

## 2. 数据结构：`AttachedFileMeta`（preview 驱动的渲染状态）

`src/stores/chat/types.ts`

```ts
export interface AttachedFileMeta {
  fileName: string;
  mimeType: string;
  fileSize: number;
  preview: string | null;
  filePath?: string;
}
```

- 核心：`preview` 的值（`string | null`）决定 UI 是显示图片缩略图，还是显示占位卡片。
- `filePath` 是缓存命中与“按引用补齐”的关键 key。

---

## 3. 本地缓存策略（helpers.ts）

### 3.1 缓存载入/落盘（localStorage）

`src/stores/chat/helpers.ts`

```ts
const IMAGE_CACHE_KEY = 'clawx:image-cache';
const IMAGE_CACHE_MAX = 100;

function loadImageCache(): Map<string, AttachedFileMeta> {
  const raw = localStorage.getItem(IMAGE_CACHE_KEY);
  if (raw) {
    const entries = JSON.parse(raw) as Array<[string, AttachedFileMeta]>;
    return new Map(entries);
  }
  return new Map();
}

function saveImageCache(cache: Map<string, AttachedFileMeta>): void {
  const entries = Array.from(cache.entries());
  const trimmed = entries.length > IMAGE_CACHE_MAX
    ? entries.slice(entries.length - IMAGE_CACHE_MAX)
    : entries;
  localStorage.setItem(IMAGE_CACHE_KEY, JSON.stringify(trimmed));
}
```

要点：
- 缓存是模块级别的 `_imageCache: Map<string, AttachedFileMeta>`（key = `filePath`）。
- `IMAGE_CACHE_MAX` 控制上限，超出则“裁剪旧条目”，避免 localStorage 无界增长。

### 3.2 缓存写入（upsert）

`src/stores/chat/helpers.ts`

```ts
function upsertImageCacheEntry(filePath: string, file: Omit<AttachedFileMeta, 'filePath'>): void {
  _imageCache.set(filePath, { ...file, filePath });
  saveImageCache(_imageCache);
}
```

这个函数会被发送端调用（见下一节），确保“历史快速加载”时仍能命中缓存。

## 4. 竞态避免：发送前先缓存图片（runtime-send-actions.ts）

`src/stores/chat/runtime-send-actions.ts` 中，在执行 `invokeIpc(...)` 之前：

```ts
if (hasMedia && attachments) {
  for (const a of attachments) {
    upsertImageCacheEntry(a.stagedPath, {
      fileName: a.fileName,
      mimeType: a.mimeType,
      fileSize: a.fileSize,
      preview: a.preview,
    });
  }
}
```

该段注释解释了为什么要“提前缓存”：
- Gateway/历史可能在 RPC 返回前就先触发 `chat.history` 刷新
- 但 history 恢复 `_attachedFiles` 时需要从缓存拿到 `preview`
- keying by `a.stagedPath`（stagedPath 会出现在 Gateway user message 文本里的 `[media attached: <path> ...]`）

## 5. 历史恢复：enrichWithCachedImages（从文本解析附件引用 + 优先走缓存）

`src/stores/chat/helpers.ts`

### 5.1 解析 `[media attached: ...]` 引用

`extractMediaRefs(text: string)` 使用正则：

```ts
const regex = /\[media attached:\s*([^\s(]+)\s*\(([^)]+)\)\s*\|[^\]]*\]/g;
```

提取结果结构：`{ filePath, mimeType }`

### 5.2 解析“原始绝对路径”（兼容工具输出文本里直接写的路径）

`extractRawFilePaths(text: string)`：
- 同时匹配 Unix（`/...` 或 `~/<...>`）与 Windows（`C:\...`）
- 只接受常见后缀（png/jpg/pdf/zip/mp4 等）生成 `{ filePath, mimeType }`

### 5.3 组装 `_attachedFiles`：命中缓存就直接带 preview，否则 preview = null

`enrichWithCachedImages(messages)` 中：
- 对 `user/assistant` 消息进行处理（且跳过已存在 `_attachedFiles` 的情况）
- 从消息文本提取 `[media attached: ...]` 与 raw file paths
- `allRefs.map(ref => { cached = _imageCache.get(ref.filePath) ... })`

当 `cached` 存在 -> `preview` 直接回填；
当不存在 -> `preview: null`（后续交给 `loadMissingPreviews` 异步补齐）。

## 6. 异步补齐：loadMissingPreviews（只为缺失 preview 发请求）

`src/stores/chat/helpers.ts` 的核心流程：

### 6.1 收集需要补齐的 filePath（去重 + 区分图片/非图片）

- 对每条 message 的 `msg._attachedFiles`
- 去重集合 `seenPaths`
- 计算 `needsLoad`：
  - 图片：`!file.preview`
  - 非图片：`file.fileSize === 0`（用来让非图片卡片有文件大小等信息）

最终得到 `needPreview: Array<{ filePath, mimeType }>`

### 6.2 IPC 获取缩略图/文件信息（按 filePath 返回）

```ts
const thumbnails = await invokeIpc(
  'media:getThumbnails',
  needPreview,
) as Record<string, { preview: string | null; fileSize: number }>;
```

### 6.3 写回到 messages 与本地缓存（触发 UI 更新）

- 遍历每条消息中 `msg._attachedFiles`
- 找到对应 `fp` 后更新：
  - `file.preview = thumb.preview`（如果有）
  - `file.fileSize = thumb.fileSize`（如果有）
  - `_imageCache.set(fp, { ...file })`
- 如果有更新则 `saveImageCache(_imageCache)`

这一步保证下一次渲染时不用重复请求。

## 7. 异步触发时机：history-actions.ts

`src/stores/chat/history-actions.ts`

在 `applyLoadedMessages(...)` 内部：
1. 先通过 `enrichWithToolResultFiles(rawMessages)`
2. 再通过 `enrichWithCachedImages(filteredMessages)`（先从缓存恢复 preview）
3. `set({ messages: finalMessages, ... })` 后
4. 调用：

```ts
loadMissingPreviews(finalMessages).then((updated) => {
  if (updated) {
    set({
      messages: finalMessages.map(msg =>
        msg._attachedFiles
          ? { ...msg, _attachedFiles: msg._attachedFiles.map(f => ({ ...f })) }
          : msg
      ),
    });
  }
});
```

要点：
- `loadMissingPreviews` 会“就地修改” `finalMessages` 里的 `_attachedFiles`
- React 需要新引用触发渲染，所以最终用 `map` 做了浅拷贝替换。

## 8. UI 渲染：ChatMessage.tsx（preview 有无决定显示）

`src/pages/Chat/ChatMessage.tsx`

用户消息区域（`isUser && attachedFiles.length > 0`）与助手消息区域（`!isUser && attachedFiles.length > 0`）都遵循同样逻辑：

- 图片附件（`file.mimeType.startsWith('image/')`）：
  - `file.preview` 有值 -> `ImageThumbnail src={file.preview} ...`
  - `file.preview` 为 null -> 显示占位 `div` + `File` 图标
- 非图片附件 -> 走 `FileCard`

因此当 `loadMissingPreviews` 异步写回 `file.preview` 后，UI 会自动从占位切换到缩略图。

## 9. 后端缩略图实现（用于理解“为什么会发请求”）

你在 helpers.ts 里看到的 IPC：`media:getThumbnails`，由：
- `electron/main/ipc-handlers.ts`
  - `ipcMain.handle('media:getThumbnails', async (_, paths) => { ... })`

它对每个 `filePath/mimeType`：
- `stat` 获取 `fileSize`
- 如果是 `image/*` -> `generateImagePreview(filePath, mimeType)`
- 返回结构：`{ [filePath]: { preview: string | null, fileSize: number } }`

另外，`src/stores/chat.ts` 旧逻辑使用 HTTP 路由：
- `electron/api/routes/files.ts`
  - `url.pathname === '/api/files/thumbnails' && req.method === 'POST'`
  - body: `{ paths: Array<{ filePath, mimeType }> }`

## 10. 推荐阅读顺序（对照源码最快）

1. 先读 `src/stores/chat/runtime-send-actions.ts` 的 `sendMessage`（找到“发送前缓存”）
2. 再读 `src/stores/chat/helpers.ts`：
   - `IMAGE_CACHE_KEY` / `_imageCache` / `upsertImageCacheEntry`
   - `enrichWithCachedImages`（从历史文本恢复 `_attachedFiles`）
   - `loadMissingPreviews`（异步补齐缺失 preview）
3. 再读 `src/stores/chat/history-actions.ts`：它如何在 `loadHistory` 后触发异步补齐并触发 re-render
4. 最后读 `src/pages/Chat/ChatMessage.tsx`：确认 UI 如何根据 `preview` 显示缩略图或占位
5. 如需理解请求是怎么生成缩略图的，再看 `electron/main/ipc-handlers.ts` 的 `media:getThumbnails`

## 11. 你可能会遇到的“差异点”（避免读错文件）

- `src/stores/chat/helpers.ts` 是当前 history 流程导入的版本（从 `history-actions.ts` 的 import 可以确认）。
- `src/stores/chat.ts` 里也包含类似实现，但它用的是 `/api/files/thumbnails`；如果你读到的不是 IPC 通道，就优先把实现锚点放回 `helpers.ts + history-actions.ts`。
